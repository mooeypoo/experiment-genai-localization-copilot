**Parameterized Profile Activity Message**
Changed from three separate keys to single parameterized message:
```javascript
// Before:

  "her": "View her activity and contributions.",
  "his": "View his activity and contributions.",
  "their": "View their activity and contributions."

// After:
```

Rationale:
- **Reduced duplication**: One message instead of three per language (7 messages vs 21 total)
- **Easier to maintain**: Single translation point, less chance for inconsistency
- **More flexible**: Can inject any pronoun form without code changes
- **Language-agnostic**: Works naturally across all language structures

## 3) Ambiguities Encountered

**Should profile.viewActivity use separate keys or single parameterized message?**
- Options considered:
  - (a) Keep three separate keys (her/his/their)
  - (b) Single parameterized message with pronoun injection (chosen)
  - (c) Completely remove gendered reference, use neutral phrasing
- Resolution: Parameterized message (option b)
- Challenge: In gendered languages, pronoun might not agree grammatically with other sentence elements
- Rationale: Showing user's chosen pronoun is more important than perfect grammar; reduces duplication significantly; easier to extend with new pronoun options

**Should we keep pluralization for non-pluralizing languages like Chinese?**
- Options considered:
  - (a) Use pipe notation with identical forms for consistency
  - (b) Use simple string for non-pluralizing languages, pipes only for pluralizing ones
  - (c) Implement custom pluralization logic per language
- Resolution: Pipe notation everywhere (option a)
- Challenge: Looks awkward in JSON to see "{count} æ¡è¯„è®º | {count} æ¡è¯„è®º | {count} æ¡è¯„è®º"
- Rationale: Consistent API across all languages; Vue i18n works the same way everywhere; no conditional logic needed in components; easier to maintain

**How to handle gendered adjective agreement in character counter?**
- Options considered:
  - (a) Use masculine form as default ("restant" in French)
  - (b) Implement full plural agreement ("restant" vs "restants")
  - (c) Rephrase to avoid agreement issues
- Resolution: Full plural agreement (option b)
- Challenge: Not all languages have same plural rules (French: 0/1 vs 2+, English: 1 vs other, Arabic: complex rules)
- Rationale: Grammatically correct messages important for professional appearance; Vue i18n supports this natively; worth extra translation effort

**Should validation messages mention field names?**
- Options considered:
  - (a) Generic: "This field is required"
  - (b) Specific: "Display name is required"
  - (c) Very specific: "The display name field cannot be empty"
- Resolution: Specific but concise (option b)
- Challenge: More specific = more translation work, but also clearer for users
- Rationale: Forms have multiple fields, users need to know which one has error; balanced specificity (include field name) with brevity (no extra articles/flourishes)

**Should error messages use first person ("We could not...") or passive voice?**
- Options considered:
  - (a) First person: "We could not find that profile"
  - (b) Passive voice: "Profile not found"
  - (c) Second person: "You entered an invalid profile"
- Resolution: Passive voice (option b)
- Challenge: Some languages (Spanish, French) prefer active voice for errors; passive can sound stiff
- Rationale: Standard pattern across web apps; more concise; translates well to most languages; front-loads important information

**Should parentheses be in translations or added via template?**
- Options considered:
  - (a) Include in translations: "({count} remaining)"
  - (b) Add via template: "(" + t('charCounter.remaining') + ")"
  - (c) Remove parentheses entirely, use different styling
- Resolution: Remove parentheses (option c)
- Challenge: Slight loss of visual grouping
- Rationale: Cleaner design; parentheses don't translate conceptually to all languages; CSS can handle visual separation; more modern UX pattern

**How to handle Arabic/Farsi complex plural rules?**
- Options considered:
  - (a) Implement full Arabic plural rules (1, 2, 3-10, 11+)
  - (b) Simplified: singular vs plural (chosen)
  - (c) No pluralization (all forms identical)
- Resolution: Simplified pluralization (option b)
- Challenge: Arabic has very complex plural rules (different forms for 0, 1, 2, 3-10, 11-99, 100+, etc.)
- Rationale: Vue i18n only supports 3 forms by default; implementing full Arabic rules requires custom pluralization functions; current bio counter shows max 160, unlikely to hit all plural categories; can enhance later if needed

## 4) Tradeoffs

**Parameterized Pronouns vs Perfect Grammar:**
- Chose parameterization over grammatical perfection
- Tradeoff: In gendered languages, sentence might not be perfectly grammatical
- Benefit: User's chosen pronoun is visible; extensible system; less translation work
- Example: French "Voir elle activitÃ©" is grammatically awkward but shows user's pronoun choice

**Consolidated Validation vs Specific Error Messages:**
- Chose to consolidate "required" and "empty" messages
- Tradeoff: Less specific error distinction (user doesn't know if field was missing vs blank)
- Benefit: Simpler mental model; less translation work; users don't care about the technical distinction

**Concise Errors vs Friendly Tone:**
- Chose concise passive-voice errors over friendly first-person messages
- Tradeoff: Less "warm" and friendly feeling, more impersonal
- Benefit: Standard web patterns; easier to scan; front-loaded with key information; shorter internationalization strings

**Uniform Pluralization vs Per-Language Optimization:**
- Chose uniform pipe notation across all languages
- Tradeoff: Chinese has redundant identical forms separated by pipes
- Benefit: Consistent API; no conditional logic; easier to maintain; works everywhere

**Removed Parentheses vs Visual Grouping:**
- Removed parentheses from counter hints
- Tradeoff: Slightly less visual separation between counter and hint
- Benefit: Cleaner design; don't need to translate punctuation semantics; more modern UX

**Simplified Arabic Plurals vs Full Linguistic Accuracy:**
- Used simplified 3-form pluralization instead of complete Arabic rules
- Tradeoff: Not fully accurate for all Arabic plural categories
- Benefit: Works with Vue i18n's built-in system; sufficient for actual use cases (small numbers); can enhance later if needed

**"Left" vs "Remaining" in English:**
- Chose "left" over "remaining"
- Tradeoff: Slightly more casual/conversational
- Benefit: Shorter, more modern UX writing style; easier to scan

**Removed Technical Terms vs Precise Descriptions:**
- Removed "ID", "body" from validation messages
- Tradeoff: Less precise for developers debugging
- Benefit: More user-friendly; simpler translations; users shouldn't see dev terminology

**Single viewActivity String vs Language-Specific Sentence Structure:**
- Used same sentence structure for all languages instead of optimizing per language
- Tradeoff: Some languages might have more natural alternative phrasings
- Benefit: Consistent UX across languages; easier to maintain; single translation point

## 5) Language Structure Implications

**Gendered Languages (French, Spanish, Arabic, Hebrew, Farsi):**

Pronoun display challenges:
- These languages assign grammatical gender to nouns and require adjective/verb agreement
- Injecting a pronoun into a sentence doesn't guarantee grammatical correctness
- Example (French): "Voir son activitÃ©" (his/her activity) - possessive "son" agrees with "activitÃ©" (feminine noun), not with person's gender
- Our approach: "Voir elle activitÃ©" - shows user's pronoun but breaks possessive agreement
- Implication: Prioritizing inclusive pronoun display over grammatical perfection

Alternative considered: Use gender-neutral rephrasing
- French: "ActivitÃ© et contributions" (activity and contributions) - no pronoun at all
- Rejected: Loses personalization; doesn't show user's choice

Validation message challenges:
- Articles must agree with field names
- "Le nom" (masculine), "La biographie" (feminine)
- Can't easily make generic "This field is required" message without knowing gender
- Solution: Keep field names in error messages to maintain proper article agreement

**Non-Gendered Languages (English, Chinese):**

Simplicity benefits:
- No grammatical gender to worry about
- Pronoun injection works naturally
- English: "View they/them activity" (grammatically correct with singular they)
- Chinese: No pronouns needed, can use "TA" (gender-neutral written form)

Pluralization challenges:
- English: Simple 1 vs many rule
- Chinese: No pluralization, use same form always
- Implication: Pipe notation in Chinese is redundant but harmless

**RTL Languages (Hebrew, Arabic, Farsi):**

Text direction complications:
- Numbers and Latin characters (like "160") don't reverse
- "45 / 160" displays correctly in RTL context
- Punctuation might need special handling (question marks, exclamation points)
- Current approach: Let browser handle BiDi algorithm

Sentence structure implications:
- Sentence order might differ from LTR languages
- Example (Arabic): Verb-Subject-Object typical, vs English Subject-Verb-Object
- Our validation messages adapt to natural Arabic structure
- "Ø§Ù„ØªØ¹Ù„ÙŠÙ‚ Ù…Ø·Ù„ÙˆØ¨" (Comment required) follows Arabic noun-adjective order

**Agglutinative Languages:**

Not currently supported, but implications for future:
- Languages like Turkish, Finnish, Japanese add suffixes to words
- Parameterized messages might require different structure
- Example: "View [user]'s activity" - possessive suffix attached in agglutinative languages
- Current approach wouldn't extend cleanly

**Languages with Complex Plural Rules:**

- 0 items
- 1 item (singular)
- 2 items (dual)
- 3-10 items (plural)
- 11+ items (also plural, but different form)

- Trade-off: Acceptable for small numbers, may sound odd for larger counts

## 9) Known Issues / Follow-ups

**Grammatical Agreement in Gendered Languages:**
- Profile activity text in French: "Voir elle activitÃ©" is grammatically incorrect
- Should be "Voir son activitÃ©" for possessive agreement
- Current approach prioritizes showing user's pronoun over perfect grammar
- Could address with language-specific message templates, but adds complexity
- Generally acceptable trade-off for inclusive representation

**Simplified Arabic Pluralization:**
- Arabic has 6 plural forms, we only use 3 (Vue i18n limitation)
- Current: singular | singular | plural
- Fully accurate would be: zero | one | two | few (3-10) | many (11-99) | other (100+)
- Works acceptably for small numbers (<100)
- Could implement custom pluralization plugin if needed

**Chinese Counter Measure Words:**
- Currently using "ä¸ª" (ge) as general measure word
- More specific measure words exist: æ¡ for comments, ä½ for people, å¼  for pages
- Current approach is acceptable but less refined
- Could customize per entity type if UX research shows importance

**Parentheses Removed from UI:**
- Character counter hint no longer visually grouped with parentheses
- Relies on CSS spacing to separate counter from hint
- Some users might find less clear
- Could add back with CSS-generated content if needed

**No Regional Variants:**
- Using es (Spanish) not es-ES or es-MX
- Using zh-CN (Simplified Chinese) not zh-TW (Traditional)
- Regional terminology differences not addressed
- Acceptable for demo, might need refinement for production

**Translation Quality:**
- Relying on machine translation plus review, not professional translators
- Some idiomatic expressions might not be perfectly natural
- Hebrew translations need native speaker verification (especially after typo fixes)
- Good enough for demo, would need professional review for production

**Pronoun Options Limited:**
- Only supporting 4 pronoun options currently
- Many languages/cultures have additional pronoun preferences
- System is extensible but translations would need updating
- Adding new pronoun requires updating all 7 language files

**Validation Message Consolidation Might Lose Information:**
- "Display name required" could mean field is null OR field is empty string
- Developers debugging might want more specificity
- Users don't care about this distinction
- Could add debug logging if developer clarity becomes important

**"Left" vs "Remaining" Tone Inconsistency:**
- English uses casual "left", other languages use formal "remaining" equivalent
- Slight tone inconsistency across languages
- Each language chose what sounds natural to native speakers
- Could enforce uniform formality level if brand voice important

**Character Counter Doesn't Account for Emoji:**
- Emoji can be multiple code units (surrogate pairs)
- JavaScript .length counts code units, not grapheme clusters
- "ðŸ‘¨â€ðŸ‘©â€ðŸ‘§" counts as 5 characters, displays as 1 emoji
- Could use Intl.Segmenter for accurate grapheme counting
- Current approach is standard JavaScript behavior, users understand

**No Translation for Avatar Alt Text:**
- Avatar alt text uses interpolated "{user.name} avatar"
- "avatar" is not translated
- Should be: `${user.name} ${t('user.avatarAlt')}`
- Low priority accessibility issue

**Comment "Unknown Author" Edge Case:**
- If author deleted, shows "Unknown author" 
- This text is translated but edge case is rare
- Might be better UX to archive deleted user info, but adds complexity

**Counter Hint Spacing:**
- Space between counter and hint: "45 / 160 5 left"
- Could look like "45 / 160 5" at quick glance
- CSS handles spacing, generally clear in context
- Could add punctuation or styling if user feedback indicates confusion

**Hebrew Pluralization Accuracy:**
- Hebrew has complex plural rules we might not be handling perfectly
- Dual form (2 items) vs plural (3+) distinction
- Current implementation may not distinguish all cases correctly
- Acceptable for current content, might need refinement from native speaker

**No Locale-Specific Validation Rules:**
- Bio character limit is 160 for all languages
- Chinese could express more in 160 characters than English
- Could have different limits per language, but adds complexity
- Uniform limit is simpler and fair enough

**French Counter Agreement Might Be Wrong for Edge Cases:**
- "0 restants" vs "0 restant" - should 0 be singular or plural in French?
- Different style guides disagree
- Current implementation uses plural for 0
- Native French speaker review recommended
